> **Development Guide for AI Coding Assistants Contributing to Personal OS**

**Version:** 1.0

# Purpose

This document defines how AI coding assistants should contribute to the Personal OS repository.

It ensures that every contribution—whether generated by ChatGPT, Codex, Claude Code, Gemini CLI, Cursor, Windsurf, Cline, or future AI tools—follows the same engineering standards, architectural principles, and development workflow.

The objective is consistency.

---

# Project Philosophy

Personal OS is **not a chatbot**.

Personal OS is **not another productivity application**.

Personal OS is an **AI Operating System** built around:

* Long-term Memory
* Decision Intelligence
* Event-Driven Architecture
* Modular Components
* AI Agents
* Capability Discovery

Every contribution should strengthen these principles.

# Architecture Rules

## Rule 1

Everything is a Component.

Examples:

* Module
* Plugin
* Agent
* Workflow
* Tool

Every component must register with the Kernel.

---

## Rule 2

Business logic belongs inside Modules.

Never implement business rules inside:

* Plugins
* Agents
* Platform
* Workflows

---

## Rule 3

Agents reason.

Agents do not own business logic.

Agents do not access databases.

Agents do not modify persistence.

---

## Rule 4

Plugins integrate external systems.

Plugins may:

* Authenticate
* Read
* Normalize
* Publish Events

Plugins must never contain business calculations.

---

## Rule 5

The Kernel owns:

* Lifecycle
* Discovery
* Registration
* Dependency Resolution
* Capability Lookup
* Runtime Orchestration

No component should bypass the Kernel.

---

## Rule 6

Communication happens through:

* Events
* Public APIs

Never access another component's database or repository directly.

---

## Rule 7

Capabilities are discovered dynamically.

Preferred:

```python
kernel.execute("finance.expense.create")
```

Avoid:

```python
FinanceService()
```

Components should depend on capabilities, not implementations.

---

## Rule 8

Memory Runtime owns user context.

Knowledge Runtime owns external knowledge.

These two concerns must remain separate.

---

## Rule 9

Platform provides infrastructure only.

Examples:

* Logging
* Configuration
* Database
* Redis
* Authentication
* AI Provider
* Storage

Platform code must not contain business logic.

---

## Rule 10

All architectural changes require an Architecture Decision Record (ADR).

Do not modify the architecture without documenting the decision.

# Coding Standards

Every implementation should:

* Use Python type hints.
* Prefer asynchronous APIs where appropriate.
* Use dependency injection.
* Keep functions small.
* Keep classes focused.
* Follow SOLID principles.
* Follow Hexagonal Architecture.
* Follow Domain-Driven Design.

Avoid:

* Global state
* Circular dependencies
* God classes
* Hidden side effects
* Framework-specific business logic

---

# Documentation Standards

Documentation is part of the implementation.

Whenever code changes:

Update:

* API documentation
* Data model
* Events
* Capabilities
* Configuration
* Workflows

Documentation should never lag behind the implementation.

---

# Testing Standards

Every implementation should include:

* Unit Tests
* Integration Tests

When appropriate:

* Contract Tests
* Event Tests
* Workflow Tests
* Performance Tests
* End-to-End Tests

Code without tests is considered incomplete.

---

# Logging Standards

Use structured logging.

Every log entry should include:

* Correlation ID
* Request ID
* Component
* Capability
* Timestamp

Never log:

* Passwords
* Secrets
* Access Tokens
* Personally Identifiable Information
* Sensitive financial information

---

# Error Handling

Errors should be:

* Explicit
* Typed
* Actionable

Avoid generic exceptions.

Domain errors should be represented using domain-specific exception types.

---

# Security

Always:

* Validate input.
* Sanitize external data.
* Use least privilege.
* Keep secrets outside source code.
* Respect authorization boundaries.

Never:

* Hardcode credentials.
* Assume trusted input.
* Expose internal implementation details.
* Log sensitive information.

---

# Performance

Optimize only after measuring.

Prefer:

* Readability
* Maintainability
* Correctness

Avoid premature optimization.

---

# AI Prompting Strategy

When implementing a feature:

1. Understand the specification.
2. Understand the architecture.
3. Understand the module boundaries.
4. Implement the smallest working change.
5. Write tests.
6. Update documentation.
7. Verify architectural compliance.

---

# Architecture Review Checklist

Before submitting any implementation, verify:

* Does it respect module boundaries?
* Does it follow the Component model?
* Does it use the Kernel correctly?
* Does it publish required events?
* Does it avoid duplicate business logic?
* Does it update documentation?
* Does it include tests?
* Does it follow naming conventions?

If any answer is **No**, revise the implementation.

---

# Commit Strategy

Use Conventional Commits.

Examples:

```text
feat(finance): implement expense aggregate

feat(kernel): add capability registry

feat(memory): support episodic memories

fix(api): validate request payload

docs(finance): update API specification

test(workflow): add retry integration tests

refactor(platform): simplify configuration loading
```

Keep commits:

* Small
* Atomic
* Reviewable

# Long-Term Evolution

When adding new functionality:

Prefer:

* New Components
* New Capabilities
* New Events
* New Workflows

Avoid modifying stable core abstractions unless required by an approved ADR.

---

# Success Criteria

A contribution is considered successful when it:

* Respects the architecture.
* Improves maintainability.
* Is independently testable.
* Is fully documented.
* Introduces no architectural drift.
* Can be understood by another contributor without additional explanation.
